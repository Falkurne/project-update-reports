Motofi infrastructure and launch roadmap

Last updated: 20 May 2026. Stack, architecture, build phases, and launch checklist for the MVP.

Decision brief Project update MVP working plan

Stack Decision

Use Vite React + Convex + Clerk + Vercel for the MVP.

This stack keeps the build close to tools already familiar to the team while giving Motofi a real backend for marketplace state, subscriptions, server-side CarJam calls, Stripe webhooks, and live dashboard updates.

Layer Choice Role
Frontend Vite React + TypeScript Dealer and broker dashboards, forms, bid review, account pages.
UI styling Tailwind CSS + small internal component set Fast app UI with consistent forms, tables, badges, modals, and empty states.
Auth Clerk Sign up, sign in, user sessions, user profile, role metadata.
Backend/database Convex Listings, bids, users, subscription status, audit events, contact unlocks, server actions.
Hosting Vercel Static frontend hosting, preview deployments, production domain.
Payments Stripe Billing Separate broker and dealer monthly subscriptions; exact amounts TBD.
Vehicle lookup CarJam API Server-side number plate lookup. Store all returned vehicle fields except valuation; odometer remains manual entry only.
Email Resend Launch emails for bid accepted, listing closed, and account/support notifications.
CRM Pipedrive Business CRM for dealer/broker sales pipeline, onboarding follow-up, and customer relationship notes.
Monitoring Vercel analytics + Convex logs + Stripe webhook logs Enough visibility for pilot launch without overbuilding observability.

System Architecture

flowchart LR user[Dealer / Broker Browser] web[Vite React App on Vercel] clerk[Clerk Auth] convex[Convex Backend] stripe[Stripe Billing] carjam[CarJam API] resend[Resend Email] pipedrive[Pipedrive CRM] user --> web web --> clerk web --> convex convex --> clerk convex --> stripe stripe --> convex convex --> carjam convex --> resend convex --> pipedrive pipedrive --> convex

Ownership rule: React renders the product, Clerk proves identity, Convex owns marketplace state and business rules, Stripe owns billing state, CarJam owns vehicle lookup, Resend owns outbound email delivery, and Pipedrive owns business CRM pipeline/follow-up context.

Core Infrastructure

Frontend App

  • Create a Vite React TypeScript app in apps/web.
  • Use Clerk React for auth and wrap the app with Clerk + Convex providers.
  • Use route-level separation for unauthenticated pages, dealer workspace, broker workspace, billing, and support/admin screens.
  • Keep the UI app-first and operational: dashboards, tables, forms, modals, filters, and status badges.

Convex Backend

Core tables:

  • users: Clerk user ID, role, company, contact details, approval status.
  • organisations: dealer yards and broker companies.
  • subscriptions: role, Stripe customer ID, subscription ID, status, current period, price ID.
  • listings: dealer/org, vehicle fields, buyer finance context, deadline, status.
  • carjamLookups: lookup payload, plate, response status, manual/support override flags.
  • bids: listing, broker/org, rate, term, conditions, status, submitted/updated timestamps.
  • contactUnlocks: listing, winning bid, dealer/broker contact snapshot, unlocked timestamp.
  • crmLinks: Convex record references mapped to Pipedrive organization/person/deal IDs.
  • auditEvents: support overrides, listing publish, bid accepted, Stripe status changes.

Critical Convex rules:

  • Only approved dealers can publish listings.
  • Only subscribed/active dealers can post paid listings once pricing is live.
  • Only approved/subscribed brokers can view live listings and bid.
  • One active bid per broker per listing.
  • Bids are semi-sealed: dealers see bids; brokers do not see competitor rates.
  • Dealer can accept exactly one bid; acceptance closes the listing and unlocks contact.
  • Contact details unlock only after accept.
  • CarJam lookup can be manually overridden by dealer or support; odometer is manual only.

Clerk Auth

  • Store the app role in Clerk public metadata or sync it into Convex after sign-up.
  • MVP roles: dealer, broker, support.
  • Require support/manual approval before a new account can create paid marketplace activity.
  • Use Convex as the source of truth for app permissions after Clerk identity is verified.

Stripe Billing

  • Create separate broker and dealer subscription products.
  • Exact monthly prices are not ready yet, so use configurable Stripe Price IDs in environment variables.
  • Stripe webhooks should update Convex subscription records.
  • Paid action gates:
    • Dealer needs active subscription before posting live listings.
    • Broker needs active subscription before accessing live listings and placing bids.
  • Do not hard-code pricing in app logic; show plan labels and status from config/data.

CarJam Integration

  • Run plate lookup through a Convex action so the API key never reaches the browser.
  • Store all returned vehicle fields except valuation.
  • Do not trust or auto-fill odometer from lookup; ask dealer to enter odometer manually.
  • If lookup fails, allow dealer manual override.
  • Add support-only override path for corrections.

Email

Use Resend for simple transactional email:

  • Dealer: new bid received.
  • Broker: bid accepted / listing closed.
  • Dealer and broker: contact unlock confirmation.
  • Support/admin: account approval or override notifications if needed.

Email is useful for pilot reliability, but the MVP should still work through in-app status if email is delayed.

Pipedrive CRM

Use Pipedrive as the business CRM, not the product database.

Pipedrive should track:

  • Dealer and broker prospects before signup.
  • Dealer yard and broker company customer records after signup.
  • Onboarding/sales stages for accounts, subscriptions, and pilot follow-up.
  • Activities for support calls, account approval follow-up, failed onboarding, and subscription renewal conversations.
  • Notes for relationship context that does not belong in the marketplace product UI.

Recommended object mapping:

  • Pipedrive Organization: dealer yard or broker company.
  • Pipedrive Person: primary contact at that company.
  • Pipedrive Deal: account onboarding/subscription opportunity, not individual Motofi listings or bids.
  • Pipedrive Activity: follow-up task for approval, payment setup, pilot check-in, support issue, or churn risk.
  • Pipedrive Note: human CRM notes and high-level product milestones.

Integration direction:

  • Convex remains the source of truth for listings, bids, subscriptions, contact unlocks, and marketplace permissions.
  • Pipedrive receives selected lifecycle events from Convex: account created, account approved, subscription active/inactive, first listing posted, first bid placed, first deal awarded.
  • Pipedrive IDs are stored in crmLinks so support/admin screens can link to the relevant CRM record.
  • For MVP, use a private Pipedrive API integration stored in Convex environment variables. Use OAuth only if Motofi later needs a multi-tenant public Pipedrive app.
  • If Pipedrive webhooks are enabled later, use them only to sync CRM fields back into Convex that the product explicitly needs, such as account owner or CRM stage. Do not let Pipedrive overwrite marketplace-critical state.

Environment And Secrets

Keep secrets out of the frontend bundle. Frontend-safe values use Vite's VITE_ prefix; all private keys belong in Convex environment variables or Vercel deployment settings that are never exposed to browser code.

Build Roadmap

Phase 0: Repo And Environment Setup

Goal: create the app foundation without implementing marketplace features.

  • Create monorepo structure and Vite app.
  • Add Convex, Clerk, Tailwind, routing, and environment variable templates.
  • Add Vercel project setup and preview deployment.
  • Move planning artifacts into docs/product, docs/technical, and docs/references.
  • Add README with local setup and required environment variables.

Acceptance:

  • App boots locally.
  • Convex dev server runs.
  • Clerk sign-in/sign-out works.
  • Vercel preview deploy succeeds.

Phase 1: Accounts, Roles, And Subscriptions

Goal: users can enter the product with the correct role and billing state.

  • Implement Clerk sign-up/sign-in.
  • Create Convex user/org profile on first login.
  • Add role selection or support-assigned role flow.
  • Add account approval status.
  • Add Stripe customer/subscription sync through webhooks.
  • Add CRM sync for new dealer/broker organizations and primary contacts.
  • Add role-based route guards and paid action guards.

Acceptance:

  • Dealer, broker, and support users land in the correct area.
  • Unapproved users cannot access marketplace actions.
  • Missing/inactive subscription blocks paid actions.
  • Stripe test webhook updates Convex subscription state.
  • New approved accounts can be linked to Pipedrive organizations/persons.

Phase 2: Dealer Listing Creation

Goal: dealers can create a live listing with CarJam lookup and buyer context.

  • Build dealer create-listing form.
  • Add plate lookup with CarJam action.
  • Store all CarJam fields except valuation.
  • Require manual odometer.
  • Include buyer income band, credit band, optional notes, location, and deadline.
  • Allow dealer manual override and support-only correction.
  • Publish listing to live.

Acceptance:

  • Dealer can create and publish a listing.
  • Failed CarJam lookup does not block posting.
  • Listing has enough buyer/vehicle context for broker bidding.
  • Only subscribed/approved dealers can publish.

Phase 3: Broker Marketplace And Bidding

Goal: brokers can browse live listings and submit structured bids.

  • Build broker live listings view.
  • Add listing detail view.
  • Add bid form with rate, term, and conditions.
  • Enforce one active bid per broker per listing.
  • Allow bid edit/withdraw while listing is live.
  • Hide competitor rates from brokers.

Acceptance:

  • Broker can browse live listings only when approved/subscribed.
  • Broker can submit, edit, and withdraw one active bid.
  • Broker cannot see competitor rates.
  • Dealer sees submitted bids in comparison view.

Phase 4: Award, Contact Unlock, And Closure

Goal: dealer accepts one broker and both sides can complete off-platform.

  • Build dealer bid comparison view.
  • Show broker/company identity before accept.
  • Accept bid with confirmation.
  • Mark listing awarded and close further bidding.
  • Create contact unlock record.
  • Show dealer/broker contact snapshots to the winning parties.
  • Notify losing brokers that listing closed.

Acceptance:

  • Only one bid can win.
  • Listing closes after award.
  • Contact unlock only happens after accept.
  • Losing brokers do not receive contact details.

Phase 5: Support/Admin And Pilot Hardening

Goal: make the pilot operable without developer intervention.

  • Add support view for account approvals.
  • Add support override for CarJam/listing corrections.
  • Add basic audit trail view.
  • Add CRM link/status panel for support users.
  • Add subscription status diagnostics.
  • Add operational empty/error states.
  • Add seed/demo data for Freddie/Mike walkthroughs.

Acceptance:

  • Support can approve accounts and inspect core records.
  • Support can open the linked Pipedrive organization/person/deal from the admin view.
  • Overrides are audited.
  • Pilot users can recover from failed lookup, inactive subscription, or withdrawn bid without confusion.

Phase 6: Launch Readiness

Goal: ship a controlled pilot.

  • Configure production Clerk, Convex, Stripe, CarJam, Resend, Pipedrive, and Vercel environments.
  • Add backup/export plan for Convex data.
  • Run end-to-end smoke tests with test dealer and broker accounts.
  • Confirm Stripe product price IDs after Freddie/Mike choose amounts.
  • Prepare pilot onboarding copy and support checklist.
  • Launch to a small controlled group of dealers and brokers.

Acceptance:

  • Production app is reachable on the final domain.
  • Dealer can post a listing with a real CarJam lookup.
  • Broker can bid.
  • Dealer can accept.
  • Contact unlock works.
  • Stripe subscription gates work in live mode.
  • Support can inspect and correct pilot issues.
  • Pipedrive receives account/subscription lifecycle events for pilot follow-up.

Launch Checklist

  • Clerk production instance configured.
  • Convex production deployment configured.
  • Vercel production project and domain configured.
  • Stripe products created for broker and dealer subscriptions.
  • Stripe Price IDs added to environment variables.
  • Stripe webhook endpoint configured and verified.
  • CarJam API credentials added to Convex environment.
  • Resend sender/domain configured.
  • Pipedrive API token/company domain configured.
  • Pipedrive pipeline and stages created for dealer/broker onboarding.
  • Support users created and tested.
  • Dealer and broker pilot accounts invited.
  • End-to-end smoke test completed.
  • Freddie/Mike approve subscription amounts and pilot copy.

Test Plan

  • Unit-test Convex business rules for publishing, bidding, withdrawing, accepting, contact unlock, and subscription gates.
  • Integration-test Stripe webhook handling with Stripe CLI in test mode.
  • Integration-test CarJam lookup success, failed lookup, dealer manual override, and support override.
  • Integration-test Pipedrive sync for account created, account approved, subscription active, and first listing posted.
  • Browser-test the critical happy path: dealer signs in, posts listing, broker bids, dealer accepts, contact unlock appears.
  • Browser-test role and subscription blocks: unapproved account, inactive subscription, broker trying to see competitor rates, losing broker trying to see contact details.
  • Smoke-test production with one dealer, one broker, and one support user before inviting pilot users.

Key Risks

  • Stripe pricing not ready: keep price IDs configurable and do not block app build on exact amounts.
  • CarJam response shape unknown: build the lookup payload storage flexibly, then map only trusted fields into listing display.
  • Broker bidding confidence: buyer income/credit band is included, but pilot feedback should decide whether more buyer fields are needed.
  • Support workload: manual/support overrides are intentional, so launch needs a simple support screen early.
  • CRM drift: Pipedrive should support sales and follow-up, but Convex must remain the product source of truth for marketplace state.
  • Scope creep into Model B: architecture should not block Model B later, but MVP remains Model A only.
