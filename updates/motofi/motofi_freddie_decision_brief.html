Decisions Made

These are the working defaults for the MVP. If any of these change, it should be an explicit product decision because each one affects the build shape.

Made

Dealer-led MVP

Start with Model A: dealers publish vehicle finance opportunities and brokers submit structured bids.

Model B and Model C stay visible as future strategy, but they do not enter the first build.

Made

CarJam day one

Number-plate lookup is included because it reduces dealer listing friction and improves data quality.

Store returned vehicle fields except valuation. Odometer remains manual entry only.

Made

Manual fallback

If lookup fails, dealers can continue with manual entry so posting is not blocked.

Support users get an override path for corrections and audit trail visibility.

Made

Buyer context included

Vehicle-only listings are too thin for useful broker bids.

Use buyer income band, credit band, optional notes, location, and deadline. Avoid a full finance application.

Made

Semi-sealed bidding

Dealers can see bid details. Brokers do not see competitor rates or an open leaderboard.

This keeps bidding cleaner and avoids a race-to-the-bottom display in the first release.

Made

Broker identity visible

Dealers see broker name and company before accepting.

That gives trust context without revealing phone/email before the dealer commits.

Made

Accept then unlock

Contact details unlock only after the dealer accepts one broker.

No pre-accept contact reveal, unlock budget, or open contact marketplace in the MVP.

Made

Stripe subscriptions

Build separate broker and dealer subscription support from day one.

Keep Stripe price IDs configurable so final prices can be dropped in later without code changes.

Made

Paid action gates

Approved dealers need an active subscription before publishing live listings.

Approved brokers need an active subscription before accessing live listings and placing bids.

Made

CRM separation

Pipedrive is the business CRM for onboarding, sales follow-up, notes, and pipeline activity.

Convex remains the product source of truth for listings, bids, billing state, contact unlocks, and permissions.

MVP Flow

The product should stay explainable in one operational path. Anything that does not help this path should wait unless the pilot proves it is necessary.

1

Account ready

Dealer or broker signs in, has a role, is approved, and has active subscription status.

2

Dealer posts

Dealer uses plate lookup, adds manual odometer, buyer bands, notes, location, and bidding deadline.

3

Brokers bid

Brokers see live listings, submit one active bid per listing, and can edit or withdraw while live.

4

Dealer awards

Dealer compares semi-sealed bids, accepts exactly one, closes the listing, and contact details unlock.

Tech Stack

The chosen stack is intentionally practical: fast to build, familiar enough to maintain, and structured enough for payments, live marketplace state, vehicle lookup, and CRM follow-up.

Layer Decision What it owns Notes for build
Frontend Vite React + TypeScript Dealer and broker dashboards, forms, bid review, billing/account pages. Keep it app-first: dense tables, forms, modals, filters, and clear status states.
Styling Tailwind CSS + small internal component set Reusable controls for forms, badges, tables, empty states, modals, and shell layout. Avoid building a large design system before the pilot proves the workflows.
Backend Convex Listings, bids, users, organisations, subscriptions, audit events, contact unlocks, and server actions. Convex is the marketplace source of truth and runs private integrations.
Auth Clerk Sign-up, sign-in, sessions, user profile, and identity handoff to Convex permissions. Use roles for dealer, broker, and support. Require approval before paid marketplace activity.
Hosting Vercel Preview deployments, production frontend hosting, and final domain routing. Keep environment values separated for development, preview, and production.
Payments Stripe Billing Broker/dealer subscriptions, customer records, subscription status, and webhook events. Use configurable price IDs. Do not hard-code subscription amounts in product logic.
Vehicle lookup CarJam API Server-side number-plate lookup and returned vehicle data storage. API key stays in Convex. Store fields flexibly until response shape is known.
Email Resend Bid received, bid accepted, listing closed, contact unlock, support/admin notifications. Email improves pilot reliability, but in-app status should still work if email is delayed.
CRM Pipedrive Dealer/broker pipeline, onboarding follow-up, relationship notes, activities, and sales stages. Sync selected lifecycle events from Convex. Do not let CRM overwrite marketplace-critical state.
Monitoring Vercel analytics + Convex logs + Stripe webhook logs Pilot-level visibility into app usage, backend errors, and billing events. Enough for launch without overbuilding observability.

Build Rules

These rules matter more than specific screens. They are the invariants the app should enforce from the first pilot release.

Only approved users can act

Dealers and brokers can sign in, but approval controls marketplace activity.

Only subscribed users can create value

Dealer posting and broker listing access/bidding are gated by Stripe subscription status once pricing is live.

One active broker bid per listing

A broker can edit the bid they have, not stack several active offers on the same listing.

Competitor rates stay hidden

Brokers do not see other broker rates while a listing is live.

Dealer accepts exactly one bid

Acceptance closes the listing, prevents later bids, and creates a contact unlock record.

Contact unlock follows award

Winning broker and dealer receive each other's contact details only after acceptance.

Support actions are auditable

CarJam overrides, account approval changes, subscription diagnostics, and listing corrections should leave a trail.

Not In The MVP

The architecture should not block these later, but they should not slow the first pilot build.

Later

Model B matching

No buyer profiles, buyer pre-qualification workflow, inventory matching, or broker-led buyer marketplace in v1.

Later

Credit checks

No bureau integration, binding lending decision, or full finance application stored in the MVP.

Later

Open auction board

No public leaderboard and no broker visibility into competitor rates.

Later

Native mobile app

Responsive web first. Mobile-specific apps wait until the pilot proves repeated usage.

Scope guard: Model B is a credible strategic lane because brokers may bring buyers, but it changes the product into an inventory and pre-qualified demand marketplace. Keep the schema flexible, but do not build Model B workflows into this MVP.

Launch Roadmap

The implementation plan is phased so Freddie and Mike can see progress at each stage instead of waiting for one large reveal.

Phase 0

Foundation

Create the Vite app, Convex project, Clerk setup, Tailwind, routing, environment templates, and Vercel preview deploy.

Phase 1

Accounts and billing

Dealer/broker/support roles, account approval, Stripe customer/subscription sync, route guards, paid action gates, and Pipedrive account links.

Phase 2

Dealer listing creation

CarJam lookup, manual odometer, failed lookup fallback, buyer bands, notes, location, deadline, and publish flow.

Phase 3

Broker marketplace

Live listings, listing detail, structured bid form, one active bid rule, edit/withdraw, and competitor rate hiding.

Phase 4

Award and contact unlock

Dealer bid comparison, accept confirmation, listing closure, winning contact snapshots, and losing broker states.

Phase 5

Support hardening

Account approval view, CarJam corrections, audit trail, subscription diagnostics, CRM links, error states, and demo data.

Phase 6

Controlled pilot

Production services, smoke test with dealer/broker/support accounts, pilot invites, onboarding copy, and support checklist.
